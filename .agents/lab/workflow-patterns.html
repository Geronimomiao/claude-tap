积木

一切都由这 三个 原语搭出来

编排 = agent（一个）× 怎么接线（parallel 有栅栏 / pipeline 无栅栏）

agent() · 一个子代理
派 1 个 agent 干一件事,可要求它按结构化 schema 返回。
parallel() · 齐头并进
一批 agent 同时跑。有栅栏 等全部跑完才继续。
pipeline() · 各走各的
每个条目独立穿过多个阶段。无栅栏 A 到第 3 步时 B 还在第 1 步。

口诀:parallel = 同步栅栏（下一步需要上一步全部结果时才用 —— 去重 / 合并 / 计数）; pipeline = 无栅栏流水（默认首选,墙钟 = 最慢那一条链,不是每步最慢之和）。下面这张动图就是这句话。

看条目 3(探 2·验 4):栅栏模式里它探完只能干等到 t=5才许开始验证;流水模式里它 t=2 立刻开验。栅栏只在"下一步真的需要上一步全部结果"时才值得付这个空等代价 —— 比如要先把所有发现去重再验证。否则默认用流水。

图 02

七种流水线形态

三个积木组合出的常见拓扑 —— 你说的"探索→验证"是其中的 ③ 和它的栅栏变体 ④

形态 ①
扇出 Fan-out
N 个 agent 各探一块,收齐。最基础的一版。
何时: 探索、独立搜集
形态 ②
扇出→汇总 Map-Reduce
收齐后加一个"综合者",把 N 份揉成一份成品。
何时: 多份结果 → 一份报告
形态 ③
流水线 Pipeline · 探索→验证
每条独立走完探索→验证,中间不设栅栏。默认首选。
何时: 每维审完立刻验 / 批量改
形态 ④
栅栏多阶段 Barrier
第二步开始前先等第一步全部到齐,中间做跨条目的活(去重)。
何时: 先合并/去重再验证 · 0 发现就整体跳过
形态 ⑤
循环收敛 Loop
不知跑几轮时循环到条件:连 2 轮无新发现 / 凑够 N / 烧到预算。
何时: 找 bug / 边界,防漏长尾
形态 ⑥
评委团 Judge Panel
N 个角度各出一版方案 → 一批评委打分 → 从赢家综合。
何时: 解法空间宽 · 设计/取舍
形态 ⑦
嵌套 Nested
一个 workflow 里再调另一个完整 workflow 当子步骤(只嵌一层)。
何时: 复用一整段已保存的编排
图 03

嫁接到"验证"那一步的四种手法

第二步具体怎么验,决定结论可信不可信

对抗式验证 Adversarial
每个结论派 N 个"唱反调" 的 agent,默认倾向反驳,多数反驳就毙掉 → 干掉"看着对其实错"。
多视角验证 Perspective
N 个验证者各用 不同镜头(正确性 / 安全 / 性能 / 能否复现)→ 抓冗余抓不到的失败模式。
多模态搜索 Multi-modal
每个 agent 用 不同方式 找(按目录 / 按内容 / 按实体 / 按时间)→ 补单一角度的盲区。
完备性批判 Critic
最后派一个 agent 专问 "还漏了什么?" → 它找到的漏洞变成下一轮的活。

规模按需要缩放:"随便看看"就几个 explorer + 单票验证;"彻底审计"就大批 finder + 3~5 票对抗验证 + 综合。
