图 01

Token 走势 & 成本:为什么会锯齿

每轮都重发整段历史 → 非线性爬升;顶到阈值被压缩砍下;再爬再砍

每轮上下文 token 压缩触发(骤降)
① 首句基线② 增量小③ 滚雪球④ 压缩骤降
不压缩(放任发散) 有压缩(被压平)

token 是锯齿,真正咬人的是下面那条累计成本线。不压缩时每轮都为越来越大的上下文付费,累计 ≈ 二次方发散;压缩把上下文封顶,成本被掰回近似线性。到第 16 轮:约 $25 vs $8.6。(真实场景还有 prompt cache,绝对金额更低、形状不变)

图 02

长上下文为何降级:U 形注意力

开头结尾看得清,中间一大坨被稀释 —— "lost in the middle"

上下文越长,越多关键信息掉进中段"迷失区"。加上注意力稀释(有限预算摊给更多 token)和训练分布外(超长靠外推),降级在压缩触发之前就已经在啃你——压缩管容量和成本,不管注意力质量。

图 03

为什么是 n²:一张"人人相看"的表

每个字的 Query 和每个字的 Key 都比一次 → n×n

行=发问的字(Q) · 列=被看的字(K) · 越深越关注 · 灰=被掩掉(看不到后面)

n 个字排成 n×n 张表,这就是 n²。下三角有效(因果掩码:一个字看不到它后面的)。上下文翻倍,这张表变 4 倍——所以越长越慢、越贵、越容易在中间走神。

图 04

本质是概率预测:下一个 token 的分布

同一个上文,模型输出的是整个概率分布,再采样其一

今天天气真 ▮ ___

模型是个无状态纯函数:文本进 → 概率分布出。采样一个字、拼回去、再算下一个,如此自回归。所谓"记忆"全在你回传的文本里;能力藏在"把分布算准"这件事里。

图 05

有状态就没问题了吗:RNN vs Transformer

你设想的"一直更新一个向量" = RNN;Transformer 反其道而行

"一直更新一个向量"就是 RNN / Mamba:无限长又便宜,但定长状态装不下全部、会忘早期。Transformer 故意抛弃它,换"每字可无损回看"——代价是 O(n²) 和有限窗口。有状态解决的是成本/重算,解决不了这个权衡。没有免费午餐。
